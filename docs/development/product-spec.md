# 제품 요구사항

## 1. 제품 정의

이 시스템은 단일 서버에서 실행되는 `S3` 호환 Object Storage다. 사용자는 AWS SDK, CLI, 또는 자체 애플리케이션을 통해 오브젝트를 업로드하고 다운로드할 수 있으며, 웹 UI를 통해 저장소 상태를 관리할 수 있어야 한다.

핵심 가치는 아래 세 가지다.

- `S3` API 호환성
- 단순한 운영 모델
- 경량 배포

## 2. 대상 사용자

- 로컬 또는 사내 환경에서 `S3` 호환 스토리지가 필요한 개발자
- 테스트 환경에서 AWS S3 대체제가 필요한 팀
- 단일 노드 백업 저장소가 필요한 소규모 서비스 운영자
- 웹 UI를 통해 오브젝트를 직접 관리해야 하는 관리자

## 3. 제품 원칙

- 단일 노드 기준에서 강한 일관성을 제공한다.
- 외부 의존성을 최소화한다.
- 운영자가 이해하기 쉬운 저장 구조를 유지한다.
- API 호환성을 우선하고, 내부 구현은 단순하게 유지한다.
- 기능 추가보다 복구 가능성과 데이터 무결성을 먼저 보장한다.

## 4. 기능 범위

### 4.1 MVP 필수 기능

- 버킷 생성, 삭제, 목록 조회, 헤더 조회
- 오브젝트 업로드, 다운로드, 삭제, 헤더 조회
- Prefix/Delimiter 기반 오브젝트 목록 조회
- Multipart Upload 시작, 파트 업로드, 완료, 중단
- Presigned URL 생성에 필요한 `SigV4` 검증
- Range 요청 처리
- 사용자 정의 메타데이터(`x-amz-meta-*`) 저장
- `ETag`, `Content-Type`, `Content-Length`, `Last-Modified` 관리
- 관리자 웹 UI
- Docker 이미지 및 `docker compose` 실행

### 4.2 제한적으로 지원할 기능

- ACL: `private`만 허용하거나 헤더를 무시하되, 문서화한다.
- Virtual-hosted-style 요청: 설정된 도메인 suffix가 있을 때만 지원한다.
- Copy Object: 동일 인스턴스 내부 복사만 우선 지원한다.

### 4.3 MVP 제외 기능

- Versioning
- Replication
- Lifecycle Rules
- Bucket Policy / IAM 전체 구현
- Server-side encryption with KMS
- Object Lock / Legal Hold
- Notification / Event Bridge 계열 기능
- Storage Class 티어링

## 5. S3 호환성 범위

초기 버전은 "표준 SDK가 붙는가"를 기준으로 범위를 잡는다. 아래 API를 우선 구현 대상으로 둔다.

### 5.1 버킷 API

- `GET Service` (`ListBuckets`)
- `PUT Bucket`
- `HEAD Bucket`
- `DELETE Bucket`
- `GET Bucket` (`ListObjectsV2`)

### 5.2 오브젝트 API

- `PUT Object`
- `GET Object`
- `HEAD Object`
- `DELETE Object`
- `COPY Object`

### 5.3 Multipart API

- `CreateMultipartUpload`
- `UploadPart`
- `CompleteMultipartUpload`
- `AbortMultipartUpload`
- `ListParts`

### 5.4 인증 및 URL

- `AWS Signature Version 4`
- Presigned `GET`
- Presigned `PUT`

### 5.5 MVP 호환 계약 요약

- `path-style` 요청을 기본 지원 방식으로 둔다.
- 기본 리전은 `us-east-1`이며, MVP는 단일 리전만 지원한다.
- 미지원 기능은 조용히 무시하지 않고 `S3` XML 오류 응답으로 거절한다.
- overwrite, `ETag`, Multipart, 에러 코드, 헤더 호환 범위는 별도 계약 문서로 고정한다.

상세 규약은 [s3-compatibility-matrix.md](./s3-compatibility-matrix.md)를 따른다.

## 6. 비기능 요구사항

### 6.1 성능

- 단일 노드에서 수천 개 수준의 버킷과 수백만 개 수준의 오브젝트 메타데이터를 처리할 수 있어야 한다.
- 대용량 파일 업로드는 메모리에 모두 적재하지 않고 스트리밍 방식으로 처리해야 한다.
- 목록 조회는 페이징 기반이어야 한다.

### 6.2 일관성

- 단일 노드에서는 쓰기 완료 직후 읽기 가능한 강한 일관성을 제공해야 한다.
- 오브젝트 업로드 완료 시점에 메타데이터와 실제 파일 상태가 불일치하지 않아야 한다.

### 6.3 내구성

- 프로세스 비정상 종료 후에도 커밋된 오브젝트는 복구 가능해야 한다.
- 미완료 Multipart 파트와 임시 파일은 정리 대상이어야 한다.
- 체크섬과 `ETag` 계산이 일관되어야 한다.
- 성공 응답은 최종 blob 영속화와 메타데이터 커밋 이후에만 반환해야 한다.

### 6.4 운영성

- 상태 확인용 `/healthz`와 `/readyz`를 제공한다.
- 기본 메트릭을 제공한다.
- 감사 로그 또는 요청 로그를 남길 수 있어야 한다.
- 데이터 디렉터리와 설정만으로 백업/복원이 가능해야 한다.
- 업그레이드 전 백업과 복원 절차가 문서화되어 있어야 한다.

## 7. 웹 UI 요구사항

웹 UI는 관리자 도구 성격으로 설계한다. 일반 사용자용 포털보다 운영 편의성이 우선이다.

### 7.1 기본 화면

- 로그인
- 대시보드
- 버킷 목록
- 버킷 상세
- 오브젝트 브라우저
- 오브젝트 업로드
- 접근 키 관리
- 시스템 상태 화면

### 7.2 대시보드 정보

- 총 버킷 수
- 총 오브젝트 수
- 총 사용 용량
- 최근 업로드/삭제 이벤트
- 현재 진행 중인 Multipart 업로드 수

### 7.3 버킷 상세 기능

- Prefix 기반 탐색
- 오브젝트 검색
- 다운로드
- 삭제
- 메타데이터 보기
- Presigned URL 발급

### 7.4 설정 화면 요구사항

- 현재 유효 설정값 조회
- 오브젝트 저장 경로, 메타데이터 DB 경로, 임시 경로 조회
- 디스크 사용량과 여유 공간 조회
- 재시작이 필요한 설정 여부 표시
- 접근 키 및 관리자 계정 설정 관리

초기 버전에서는 웹 UI가 호스트 파일시스템을 직접 탐색하거나 Docker 볼륨을 생성하는 기능은 제공하지 않는다. 경로 변경은 제한적으로만 허용하고, 변경 시 서버 재시작이 필요하다고 본다.

## 8. 인증과 권한 모델

### 8.1 API 인증

- `Access Key ID` + `Secret Access Key` 기반
- `AWS SigV4` 서명 검증
- 루트 관리자 키 1세트와 추가 서비스 키 여러 개를 발급 가능하게 설계

### 8.2 UI 인증

- 관리자 계정 기반 세션 로그인
- 비밀번호는 평문 저장 금지
- 세션 쿠키는 `HttpOnly`, `Secure` 옵션을 지원

### 8.3 권한 모델

초기 버전은 복잡한 정책 엔진을 두지 않는다.

- 루트 관리자: 모든 권한
- 서비스 계정: 전체 스토리지 사용 권한 또는 향후 버킷 단위 제한 확장 가능

### 8.4 보안 부트스트랩과 수명주기

- 기본 관리자 계정이나 기본 루트 키를 제공하지 않는다.
- 빈 메타데이터 DB에서는 bootstrap 절차가 끝나기 전까지 `setup-required` 상태로 동작한다.
- 초기 관리자와 초기 루트 접근 키는 bootstrap 입력으로만 받고, 이후에는 DB에 저장한다.
- 저장된 secret는 다시 평문으로 재표시하지 않는다.
- 키 회전, 세션 정책, 감사 로그 범위는 별도 보안 문서로 고정한다.

상세 규약은 [security-model.md](./security-model.md)를 따른다.

## 9. 배포 요구사항

### 9.1 실행 형태

- 로컬 바이너리 실행
- Docker 단일 컨테이너 실행
- `docker compose` 기반 개발 및 운영 실행

### 9.2 필수 설정

- HTTP Listen Address
- 데이터 디렉터리 경로
- 메타데이터 DB 경로
- Multipart 임시 경로
- 로그 경로
- 인스턴스 마스터 키
- 단일 리전 이름
- 초기 부트스트랩용 관리자 계정 정보 또는 bootstrap 수단
- 초기 부트스트랩용 루트 Access Key / Secret Key
- 외부 공개 URL 또는 도메인 suffix

### 9.3 설정 방식

- 설정 파일과 환경변수를 모두 지원해야 한다.
- 환경변수는 설정 파일보다 우선한다.
- `.env`는 로컬 개발 및 `docker compose` 변수 주입용으로 사용 가능하다.
- 복잡한 구조 설정은 설정 파일이 기준이 된다.
- 환경변수로 오버라이드된 값은 UI에서 읽기 전용으로 보여야 한다.
- 부트스트랩 자격 증명은 영구 설정 파일의 일부가 아니다.

### 9.4 Docker 기준

- 데이터와 DB는 볼륨으로 분리 가능해야 한다.
- 컨테이너 재시작 시 상태가 유지되어야 한다.
- 헬스체크가 가능해야 한다.
- 컨테이너가 사용할 경로는 Docker 볼륨 또는 바인드 마운트로 먼저 연결되어야 한다.
- 웹 UI는 이미 마운트된 컨테이너 내부 경로만 다룰 수 있다.

## 10. 운영 가정

- TLS 종료는 리버스 프록시 또는 인그레스에서 처리할 수 있다.
- 시스템 시간은 정확해야 한다. `SigV4` 검증은 시간 편차에 민감하다.
- 단일 노드 디스크 장애는 시스템 전체 장애로 간주한다.
- 프록시 헤더는 명시적으로 신뢰하도록 설정한 경우에만 사용한다.

## 11. 성공 판정 기준

- AWS CLI가 커스텀 엔드포인트로 기본 파일 작업을 수행한다.
- `boto3`와 AWS SDK for Go에서 업로드/다운로드/리스트 작업이 통과한다.
- 웹 UI에서 버킷 생성, 파일 업로드, 삭제가 가능하다.
- `docker compose up` 후 별도 수동 작업 없이 초기 실행 가능하다.
- 재시작 후 메타데이터와 오브젝트 파일이 일관되게 유지된다.
- 백업본으로부터 복원 후 샘플 오브젝트 조회가 가능하다.

## 12. 열린 결정 사항

다음 항목은 구현 시작 전에 확정하면 좋다.

- 웹 UI 프런트엔드를 `React`로 할지 더 단순한 정적 템플릿 방식으로 할지
- Copy Object와 ACL 헤더를 어디까지 허용할지
- Virtual-hosted-style 지원을 첫 버전에 포함할지
- 웹 UI에서 저장 경로 변경을 어디까지 허용할지
- 다중 디스크를 MVP에 포함할지, 후속 기능으로 둘지
